# D0.1 — 컨벤션 주입 정합성

> M0 gate 문서. 강제규약 정본 위치 확인, 에이전트 규약 중복 0 검증, CLAUDE.md 미로드 증명 절차.

---

## 개요

Quartermaster 에이전트가 올바른 규약을 주입받고 있음을 보장하는 정합성 검증 문서.

핵심 원칙:
1. **강제규약 정본은 단 1곳** — 중복 없음
2. **에이전트는 정본을 참조** — 본문에 규약 내용을 복붙하지 않음
3. **CLAUDE.md는 Quartermaster 규약을 포함하지 않음** — 전역 사용자 설정과 분리

---

## 강제규약 정본 1개: manifest-schema SKILL.md

**정본 위치**: `skills/manifest-schema/SKILL.md` (또는 `.claude-plugin/skills/manifest-schema/SKILL.md`)

이 파일이 Quartermaster 매니페스트 작성 규약의 **유일한 출처(SSOT)**다.

정본에 포함되어야 할 내용:
- `manifest.schema.json` 필드별 사용법
- key 명명 규칙 (`^[a-z][a-z0-9]*(\.[a-z][a-z0-9]*)*$`)
- 필수 필드 (key / name / component)
- 허용 component ID 목록 (catalog.json 참조)
- 거부 코드별 수정 방법

**정본 로드 방법** (Claude Code hook):
```
PreWrite hook → 매니페스트 작성 시
  → skill: manifest-schema 자동 로드
  → 에이전트가 규약 숙지 후 작성
```

## 에이전트 규약 본문 중복 0 검증

**원칙**: 에이전트 파일(`agents/*.md`)은 규약 내용을 본문에 포함하지 않고, SKILL.md를 참조한다.

### grep 검증법

다음 명령으로 에이전트 파일에 key 패턴이 직접 기술되어 있는지 검사:

```sh
# 에이전트 파일에 key 정규식이 하드코딩되어 있으면 중복 위반
grep -r "a-z][a-z0-9]" agents/
# 결과가 없어야 함 (PASS) — 패턴은 schema에만 존재해야 함

# 에이전트 파일에 거부 코드가 나열되어 있으면 중복 위반
grep -rE "SCHEMA_VIOLATION|CATALOG_UNKNOWN_COMPONENT|DUPLICATE_KEY|MISSING_REQUIRED|OVERRIDABLE_VIOLATION|INVALID_KEY_FORMAT" agents/
# 결과가 없어야 함 (PASS) — 거부 코드는 CONTRACT.md/SKILL.md에만 존재해야 함

# 에이전트 파일에 component 목록이 하드코딩되어 있으면 중복 위반
grep -r "Panel_Base\|Btn_Base\|ScrollList" agents/
# 결과가 없어야 함 (PASS) — 컴포넌트 목록은 catalog.json 참조
```

CI에서 이 검증을 자동화하는 방법 (선택):
```yaml
- name: Check no rule duplication in agent files
  run: |
    if grep -rqE "SCHEMA_VIOLATION|CATALOG_UNKNOWN_COMPONENT" agents/ 2>/dev/null; then
      echo "FAIL: Rule duplication found in agents/"
      exit 1
    fi
    echo "PASS: No rule duplication"
```

### 허용 예외

에이전트 파일이 SKILL.md를 참조하는 링크/지시문은 중복이 아니다:
```markdown
# 허용 (참조)
규약: manifest-schema SKILL.md 참조

# 불허 (중복)
key 형식: ^[a-z][a-z0-9]*(\.[a-z][a-z0-9]*)*$
```

## CLAUDE.md 미로드 증명 절차

**원칙**: 사용자의 전역 `CLAUDE.md` (`~/.claude/CLAUDE.md`)는 Quartermaster 규약을 포함하지 않는다. Quartermaster 규약은 플러그인 스코프에서만 로드된다.

### 증명 방법

**방법 1 — 파일 내용 검사**:
```sh
# 전역 CLAUDE.md에 Quartermaster 규약이 없음을 확인
grep -c "quartermaster\|QmManaged\|manifest.schema\|stableKey" ~/.claude/CLAUDE.md || echo "0 matches (PASS)"
# 출력이 0 또는 "0 matches (PASS)"이어야 함
```

**방법 2 — 플러그인 스코프 격리 확인**:
```sh
# Quartermaster 규약은 플러그인 디렉토리에만 존재해야 함
find ~/.claude -name "CLAUDE.md" | xargs grep -l "quartermaster" 2>/dev/null
# 출력이 없어야 함 (PASS)

# 플러그인 내 CLAUDE.md는 허용
find .claude-plugin -name "CLAUDE.md" | xargs grep -l "quartermaster" 2>/dev/null
# 출력 있어야 함 (플러그인 스코프 규약 확인)
```

**방법 3 — Claude Code 로드 순서 추적**:

Claude Code는 다음 순서로 컨텍스트를 로드한다:
```
1. ~/.claude/CLAUDE.md         (전역 — Quartermaster 규약 없음)
2. .claude/CLAUDE.md            (프로젝트 — 게임 프로젝트 규약)
3. .claude-plugin/SKILL.md     (플러그인 — Quartermaster 규약 여기서 로드)
4. hook 트리거 시 skill 로드   (동적 — manifest-schema SKILL.md)
```

Quartermaster 규약은 3번/4번에서만 로드됨. 전역(1번)과 프로젝트(2번)에 혼입되지 않음.

**증명 체크리스트**:

| 검사 항목 | 명령 | 기대 결과 |
|----------|------|----------|
| 전역 CLAUDE.md에 QM 규약 없음 | `grep -c quartermaster ~/.claude/CLAUDE.md` | 0 |
| 전역 CLAUDE.md에 stableKey 없음 | `grep -c stableKey ~/.claude/CLAUDE.md` | 0 |
| 플러그인 SKILL.md 존재 | `ls skills/manifest-schema/SKILL.md` | 파일 존재 |
| agents/ 파일 규약 중복 없음 | 위 grep 검증법 | 0 matches |

## 규약 주입 플로우 다이어그램

```
사용자 요청: "RosterScreen 매니페스트 작성해줘"
        ↓
Claude Code PreWrite hook 트리거
        ↓
skill load: manifest-schema/SKILL.md  ← 규약 정본 (단 1회 로드)
        ↓
에이전트가 규약 숙지 (SSOT)
        ↓
매니페스트 초안 생성 (L1)
        ↓
Write 호출
        ↓
L2 hook: validate-manifest.mjs 실행
        ↓
ok:true → Write 허용
ok:false → Write 차단 + 에러 표시
```

**중복 없음**: 규약이 SKILL.md 1곳에만 존재하므로, 규약 변경 시 SKILL.md만 수정하면 된다. 에이전트/hook/CI 모두 동일 정본을 참조하므로 불일치 불가.
