# D0.2 — 검증 3계층 (Validation 3-Tier)

> M0 gate 문서. CONTRACT.md §검증 계약 구현 명세.

---

## 개요

Game UI Studio는 매니페스트 유효성을 3개 계층에서 검증한다. 각 계층은 독립적으로 작동하며, 상위 계층이 하위 계층을 보완한다.

```
L1 (설득)     → Claude Code 에이전트가 작성 시 실시간 안내
L2 (Write hook) → 파일 저장 시 차단 (SSOT: validate-manifest.mjs)
L3 (CI)       → 커밋/PR 시 전수 재검증 (우회경로 차단)
```

---

## L1 — 설득 계층 (Persuasion)

**담당**: Claude Code 에이전트 (SKILL.md 규약)  
**시점**: 매니페스트 작성 중 (Write 전)  
**방식**: 에이전트가 스키마 규약을 숙지하고 유효한 JSON을 생성  
**강제력**: 없음 (soft) — 에이전트가 잘못된 파일을 작성할 수 있음

```
에이전트 → 매니페스트 초안 작성
       ↓
  규약 준수 확인 (L1)
       ↓
  Write 호출
```

**L1이 못 잡는 것**:
- 에이전트 규약 미준수
- MCP를 통한 직접 파일 쓰기 (에이전트 우회)
- 사람이 직접 JSON 수정

---

## L2 — Write Hook 계층 (Gate)

**담당**: Claude Code Write hook  
**시점**: `.manifest.json` 파일 저장 직전  
**방식**: `node runtime/validate-manifest.mjs` 실행  
**강제력**: 강함 — exit code 2 시 Write 차단

```
Write 호출
    ↓
hook: node runtime/validate-manifest.mjs <file>
    ↓
  exit 0 → Write 허용
  exit 2 → Write 차단 (에러 메시지 표시)
```

**SSOT**: L2와 L3 모두 `runtime/validate-manifest.mjs`의 동일 함수를 호출.  
검증 로직이 한 곳에 있어 L2와 L3가 동일한 기준으로 판정한다.

**L2가 못 잡는 것**:
- **MCP 직접조립 우회**: MCP 서버가 `validate-manifest.mjs`를 거치지 않고 직접 파일을 조립하는 경우
- Claude Code 외부에서의 파일 수정 (git checkout, 다른 도구)
- L2 hook이 비활성화된 환경

---

## L3 — CI 계층 (Full Coverage)

**담당**: GitHub Actions (`.github/workflows/ci.yml`)  
**시점**: 커밋 push / PR 생성 시  
**방식**: `test/fixtures/manifests/*.manifest.json` 전수 재검증  
**강제력**: 강함 — CI RED 시 merge 차단 (branch protection rule 필요)

```
git push / PR
    ↓
CI: node --test test/unit/   (단위 테스트)
    ↓
CI: L3 전수검증 스텝         ← 핵심
    ↓
  모든 *.manifest.json 재검증
  valid → exit 0
  invalid fixtures → 올바르게 거부 확인
  exit 0 → CI GREEN
  exit 2 → CI RED (merge 차단)
```

### L3 전수검증 로직

```javascript
// .github/workflows/ci.yml L3 스텝 핵심 로직
for (const file of manifestFiles) {
  const isInvalid = file.startsWith('invalid-');
  const result = validateManifest(manifest, catalog);

  if (isInvalid && result.ok)    → FAIL (거부되어야 할 파일이 통과)
  if (!isInvalid && !result.ok)  → FAIL (유효한 파일이 거부됨)
  else                           → PASS
}
```

## MCP 우회경로와 L3의 역할

**우회 시나리오**:

```
MCP 서버 (예: Unity MCP) → Assets/UI/*.prefab 조립
                         → manifest 파일을 직접 생성 (L2 hook 없음)
```

이 경로에서 L2 hook은 트리거되지 않는다. 생성된 manifest가 저장소에 커밋되면 L3가 이를 감지한다.

**L3 우회 픽스처 검증 방법**:

1. MCP가 생성한 manifest 파일이 `test/fixtures/manifests/` 또는 매니페스트 경로에 커밋됨
2. CI L3 스텝이 해당 파일을 `validateManifest(manifest, catalog)`로 재검증
3. 유효하지 않으면 CI RED → merge 차단

**우회 픽스처 예시**:

`test/fixtures/manifests/invalid-unknown-component.json`:
- MCP가 카탈로그에 없는 컴포넌트를 사용해 조립한 시나리오를 재현
- L2는 이 파일 저장을 막았어야 하지만, MCP 직접 경로는 L2를 우회
- L3가 CI에서 이를 `CATALOG_UNKNOWN_COMPONENT` 코드로 잡아 RED 반환

## 책임 경계 요약

| 계층 | 담당 | 시점 | 강제력 | 못 잡는 것 |
|------|------|------|--------|-----------|
| L1 | 에이전트 규약 | 작성 중 | 없음 | MCP 우회, 사람 직접 수정 |
| L2 | Write hook | 저장 시 | 강함 | MCP 우회, 외부 도구 |
| L3 | CI 전수검증 | 커밋/PR | 강함 (merge 차단) | Unity 환경 필요 항목 |

**결론**: L1+L2+L3 삼중 방어로 어떤 경로로 매니페스트가 저장소에 들어오더라도 CI에서 최종 차단한다.
