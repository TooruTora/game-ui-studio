v0.3 · 책임 경계

파이프라인은 프리팹에서 멈춘다

v0.3의 결정: 코드 생성을 뺀다. 파이프라인은 베이스 프리팹을 조립해 화면 프리팹을 만드는 데까지만 책임지고, View·로직 코드는 토라가 정해진 구조에 맞춰 직접 짠다. 코드 생성/머지가 통째로 빠지면서 멱등 보존은 프리팹에만 적용되어 단순해진다.

파이프라인 (자동)

  • 매니페스트 설계·스키마 검증
  • 웹 프리뷰 렌더 / 검수 루프
  • 베이스 프리팹 인스턴스화·계층 구성
  • 레이아웃·콘텐츠 오버라이드 적용
  • 안정 키로 멱등 재조립 (프리팹)

토라 (수동)

  • MVC View MonoBehaviour 작성
  • 프리팹에 View 부착·참조 와이어링
  • FSM·로직·연출 코드
  • 구조 컨벤션 준수 (CLAUDE.md)
왜 깔끔해지나 — 코드 생성을 빼면 partial 분리·컴파일 자가수정 같은 까다로운 장치가 전부 불필요해진다. 사람이 짠 .cs는 애초에 파이프라인이 만들지도 건드리지도 않으니 보존이 자명하고, 멱등 머지는 프리팹 오버라이드 한 종류만 다루면 된다.
목표

더 적은 부품으로 닫는 루프

자연어로 화면을 요청하면 AI가 레이아웃을 설계하고, 그 매니페스트를 브라우저에서 즉시 렌더해 조립 전에 눈으로 검수한다. 승인된 매니페스트만 Unity MCP로 넘겨 UGUI 화면 프리팹을 조립한다. 코드는 사람 몫 — 파이프라인은 일관된 이름과 깔끔한 계층의 프리팹을 넘겨주고, 토라는 그 위에 구조대로 View를 얹는다. 재조립해도 사람이 부착한 컴포넌트와 수동 오버라이드는 살아남는다.

설계 원칙

다섯 개의 축

A1

매니페스트 단일 소스

JSON 하나가 유일한 진실. 웹 프리뷰와 Unity는 그걸 푸는 두 소비자일 뿐이다. 살아남는 건 매니페스트, 나머지는 산출물.

A2

단방향 방화벽

프리뷰는 매니페스트를 받아 그리기만 하는 폐기형 산출물. 좌표·스케일을 Unity로 역수입하지 않는다 — 역방향이 비결정성의 출처이기 때문.

A3

카탈로그 그라운딩

AI는 카탈로그에 등재된 component만 쓸 수 있다. 서드파티 에셋 경로·필드를 지어내는 환각을 스키마 검증 hook으로 차단.

A4

설계 / 미감 분리

레이아웃 결정(계층·앵커·간격)은 AI에게 위임. 픽셀 단위 미감은 사람이 웹 프리뷰로 빠르게 반복.

A5

멱등 프리팹 보존 + 코드 무간섭

파이프라인이 만지는 건 프리팹뿐. 재조립은 안정 키로 관리 오버라이드(앵커·치수·콘텐츠)만 재적용하고, 사람이 부착한 View 컴포넌트·참조·수동 오버라이드는 보존한다. 코드 파일은 애초에 생성하지 않으니 영원히 사람 소유. "조금 고쳐서 다시"가 안전한 연산이 된다.

데이터 흐름

검수 루프 + 보존 조립

NG 반복은 웹 프리뷰 단에서만 돈다. Unity 조립은 시각 승인 뒤, 그리고 이후의 모든 재조립은 사람의 손길을 보존하며 돈다.

1

요청 /ui-design

"용병 로스터 화면. 좌측 리스트, 우측 상세, 하단 모집 버튼."

2

설계 → 매니페스트 v1 ui-designer 에이전트

카탈로그에 있는 component만 골라 계층·앵커·간격을 JSON으로 출력. 각 element에 안정 키 부여. 스키마 검증 통과 필수.

3

웹 프리뷰 렌더 web-preview 스킬

매니페스트를 브라우저에서 즉시 렌더. flexbox≈LayoutGroup, absolute+anchor≈RectTransform로 UGUI 시맨틱을 모사. 토라가 바로 본다.

4

검수 사람

OK, 또는 "리스트 너무 좁아, 상세 패널 줄여" 같은 피드백.

↺

NG → 재설계 → 3으로

매니페스트만 수정해 프리뷰를 다시 그린다. Unity는 건드리지 않는다.

5

프리팹 조립 /ui-assemble · Unity MCP

승인된 매니페스트로 베이스 프리팹을 인스턴스화하고 계층을 구성, 레이아웃·콘텐츠 오버라이드를 적용해 화면 프리팹으로 저장. 코드는 만들지 않는다.

6

핸드오프 → 코드 작성 사람

토라가 프리팹 위에 MVC View를 붙이고 참조를 와이어링, FSM·로직·연출을 직접 짠다. 일관된 오브젝트 이름이 이 작업을 빠르게 만든다.

7

재조립 = 프리팹 보존 머지 멱등

매니페스트를 고쳐 다시 조립하면, 안정 키로 add/update/remove를 판정. 관리 오버라이드만 재적용하고 사람이 부착한 컴포넌트·참조·수동 오버라이드는 남긴다.

멱등 조립 — A5 상세

프리팹만 덮어쓰고, 나머지는 둔다

파이프라인은 프리팹의 관리 오버라이드만 소유한다. 사람이 부착한 컴포넌트·참조·수동 변경은 안정 키 기반 머지로 보존되고, 코드 파일은 영역 밖이라 애초에 닿지 않는다.

코드는 영역 밖 — 매니페스트는 레이아웃만

파이프라인은 .cs를 만들지 않고, 매니페스트엔 상호작용 정보(콜백 매핑 등)도 두지 않는다. 조립이 name으로 오브젝트를 일관되게 명명해 주는 것이 전부 — 그 일관된 이름 덕에 토라는 View를 작성하고 인스펙터에서 바로 와이어링한다.

// 토라가 직접 작성 — 파이프라인은 이 파일을 만들지도 건드리지도 않음
public class RosterView : MonoBehaviour {
  [SerializeField] Button btn_Recruit;   // 프리팹의 Btn_Recruit 드래그
  void Awake() {
    btn_Recruit.onClick.AddListener(OnRecruitClicked);  // 연결도 직접
  }
  void OnRecruitClicked() { /* 로직·연출 */ }
}

프리팹 — 안정 키 머지

매니페스트 각 element의 key(안정 GUID/path)로 기존 인스턴스를 식별한다. 재조립은 파이프라인이 관리하는 오버라이드(앵커·치수·콘텐츠)만 재적용하고, 사람이 에디터에서 추가한 자식·부착 컴포넌트·오버라이드는 건드리지 않는다.

대상재조립 시 동작소유
관리 오버라이드키 매칭으로 재적용 (앵커·치수·콘텐츠)파이프라인
부착 컴포넌트보존 (View MonoBehaviour 등)사람
참조 와이어링보존사람
수동 오버라이드·자식보존 (머지)사람
코드 파일 (.cs)영역 밖 — 미생성·미접촉사람
키 없는 매니페스트 element신규 추가파이프라인
매니페스트에서 사라진 키제거 (확인 후)파이프라인
회색지대 규칙 — 파이프라인 관리 영역(앵커·치수·콘텐츠)을 사람이 인스펙터에서 직접 바꾸면, 다음 재조립이 매니페스트 값으로 되돌린다. 그래서 레이아웃 변경은 항상 매니페스트에서, 컴포넌트 부착·와이어링·연출은 사람 영역에서. 이 경계가 보존의 핵심.
매니페스트 스키마

두 소비자의 공통 언어

웹 프리뷰 종속도, UGUI 종속도 아닌 중립 레이아웃 표현. 양쪽에 어댑터를 두고 component ID로 실체를 풀며, key로 멱등 머지를 건다.

{
  "screen": "RosterScreen",
  "refResolution": [1920, 1080],
  "elements": [
    {
      "key": "panel.roster", // 멱등 머지 키
      "name": "Panel_Roster", "component": "Panel_Base",
      "anchor": "stretch", "margin": [40,40,40,40],
      "children": [
        { "key": "list.mercs", "name": "List_Mercs",
          "component": "ScrollList", "layout": "vertical", "spacing": 8 },
        { "key": "panel.detail", "name": "Detail",
          "component": "Panel_Base", "anchor": "right", "width": 600 }
      ]
    },
    { "key": "btn.recruit", "name": "Btn_Recruit", "component": "Btn_Base",
      "anchor": "bottom-right", "label": "Recruit" }
  ]
}
필드역할누가 쓰나
key안정 식별자. 재조립 시 add/update/remove 매칭의 기준멱등 머지
component카탈로그의 부품 ID. 실체(프리뷰 노드/프리팹)는 어댑터가 푼다프리뷰 + Unity
anchor / margin추상 앵커·여백. 양쪽이 공통으로 이해프리뷰 + Unity
layout / spacing레이아웃 그룹 종류와 간격프리뷰 + Unity
width / height고정 치수 (선택)프리뷰 + Unity
label / iconSlot부품 오버라이드 값프리뷰 + Unity
카탈로그

양면 사전

component ID 하나가 프리뷰 쪽 "어떻게 보이나"와 Unity 쪽 "어떤 프리팹인가"를 동시에 가리킨다. 플러그인이 1회 스캔으로 생성하고, 이후 모든 설계가 여기 등재된 것만 참조한다.

{
  "Btn_Base": {
    "preview": { "el": "button", "cls": "btn-base" },  // 웹 렌더용
    "unity": "Assets/UI/Prefabs/Btn_Base.prefab",
    "overridable": ["label", "iconSlot"]
  },
  "Panel_Base": { "preview": { "el": "div", "cls": "panel-base" },
    "unity": "Assets/UI/Prefabs/Panel_Base.prefab" }
}
왜 더 중요해졌나 — 시각적 출발점을 외부 도구에서 뺐기 때문에, 카탈로그가 곧 AI의 "쓸 수 있는 부품 목록"이자 프리뷰의 렌더 규칙이다. 프리뷰 쪽 매핑은 가벼운 CSS 클래스 한 줄로 충분 — 베이스 프리팹의 9-slice 느낌만 대략 흉내내면 된다.

구현 로드맵

의존성 순서대로, MVP 먼저

M1~M3가 핵심 가설("자연어→매니페스트→쓸만한 레이아웃")을 싸게 검증하는 MVP. 어려운 엔지니어링인 멱등 머지(M4)는 가설이 선 다음에 친다.

M1

매니페스트 스키마 + CLAUDE.md 룰 + 손 검증

공통 언어를 확정하고, 실제 화면 2~3개(로스터·인벤토리·모달)를 손으로 매니페스트로 적어본다. 스키마가 표현 못 하는 게 여기서 다 드러난다.

의존: 없음 (기반)
M2

카탈로그 생성 (catalog-sync)

Unity MCP로 프리팹/스프라이트 폴더를 스캔해 component 인벤토리 + 프리뷰 CSS 매핑 .md/.json 생성.

의존: M1 · 사전 준비 — 베이스 프리팹
M3

경량 웹 프리뷰 (web-preview)

매니페스트 → HTML/canvas. 싸고 빠른 검수 게이트로 스키마 표현력을 즉시 검증. 여기까지가 최소 검증 루프.

의존: M1, M2
M4

멱등 프리팹 조립 (unity-assemble)

매니페스트 → 베이스 프리팹 인스턴스·계층·오버라이드. 안정 키 머지로 관리 오버라이드만 재적용, 부착 컴포넌트·수동 변경 보존. 코드 생성은 없음. 가장 어려운 부분.

의존: M1, M2 · Unity MCP
M5

루프 통합 + 핸드오프 규약 + 보존 검증

/ui-design end-to-end 연결, hook 결합, 오브젝트 네이밍·bindTo 와이어링 규약 확정(사람이 코딩하기 좋게), 재조립 보존 회귀 테스트(View 부착·와이어링 후 재조립→유지 확인).

의존: M3, M4
사전 준비물

파이프라인 밖에서 한 번 갖춰둘 것

Unity MCP 설치 — CoplayDev/unity-mcp는 Unity 버전 무관·무료(MIT), 패키지 매니저 git URL로 설치. Unity 6를 쓰면 공식 MCP Server도 선택지(구독 필요, AI 크레딧은 미소모).

상용 PNG 팩 → 베이스 프리팹 1회 굽기 — 임포트 세팅(Point/압축 없음 또는 Bilinear), 9-slice 보더, Sprite Atlas를 잡아 Btn_Base · Panel_Base 등을 만든다. 이게 카탈로그의 Unity 측 실체. PNG 노가다는 여기서 한 번만.

베이스 프리팹의 프리뷰 CSS 매핑 — 각 베이스의 대략적 외형(배경·테두리·패딩)을 가벼운 CSS 클래스로 한 번 정의. 웹 프리뷰가 이걸 쓴다. 픽셀 정확이 아니라 "형태 알아볼 정도"면 충분.

프로젝트 컨벤션 문서 — CLAUDE.md에 들어갈 앵커·오브젝트 네이밍·MVC 구조·안정 키 규칙. 사람이 코딩할 구조와 파이프라인의 명명이 여기서 만난다.

매니페스트의 성격 — 거의 닫힘

A5(멱등 보존)를 택하면서 답이 명확해졌다. 매니페스트가 진실이고 레이아웃 편집은 매니페스트에서(escape hatch), 컴포넌트 부착·와이어링·연출은 사람 영역에서. 남은 건 매니페스트의 가독성 — 주석·포맷을 사람이 읽기 좋게 갈지의 취향 결정뿐.

와이어링 자동화 — 안 함 (결정됨)

파이프라인은 와이어링에 일절 관여하지 않는다. 오브젝트 이름은 조립에서 공짜로 나오고, 그 외 참조 연결·콜백은 100% 사람 코드 몫. 따라서 매니페스트는 bindTo 같은 상호작용 필드 없이 순수 레이아웃/구조만 표현한다. 거동은 전부 코드에 산다.

프리뷰 충실도 갭. 여전히 100%는 아니다. 다만 웹 렌더러를 UGUI 시맨틱에 맞춰 줄일 수 있어 Figma보다 유리. 최종 미감은 Unity 실물에서 확인 — 프리뷰는 "깨진 레이아웃"을 싸게 잡는 게이트다.
프리뷰 렌더러 유지보수. 스키마가 바뀌면 web-preview도 같이 갱신해야 한다. 매니페스트를 단순하게 유지할수록 이 비용이 작다.
멱등 머지의 회색지대. 사람이 파이프라인 관리 영역(앵커·치수·콘텐츠)을 인스펙터에서 바꾸면 재조립이 되돌린다. 경계 규칙(레이아웃=매니페스트, 부착·와이어링=사람)을 CLAUDE.md에 명문화해 사고를 막는다.
카탈로그 동기화. PNG/프리팹을 추가하면 catalog-sync를 다시 돌려야 한다. 안 하면 AI가 새 부품을 모른다.
AI 환각. 카탈로그 밖 component 참조 시도 — 스키마 검증 hook으로 저장 단계에서 차단.
